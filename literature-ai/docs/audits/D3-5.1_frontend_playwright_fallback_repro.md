## 3. 问题说明
- **npm 不在 PATH**：在此 Agent / Shell 运行环境中，`npm` 无法被直接调用。
- **bundled Playwright fallback 可运行**：通过直接调用 `node.exe` 以及 Playwright CLI，可以在没有 `npm` 的情况下运行前端测试。
- **临时依赖问题**：但如 D3-5 中所记录，直接运行 fallback 命令需要指定由于 pnpm layout 导致的 `NODE_PATH`，并且还需要一个临时、非仓库的 `@playwright/test` shim 目录才能让测试脚本成功 `require('@playwright/test')`。
- 该问题纯粹属于本地工具链可复现性（Reproducibility）问题，不代表 D3 safety contract 或任何业务逻辑存在缺陷。

## 4. 本轮处理
- **只更新 Docs**：本轮尝试编写自动绕过 `npm` 限制并动态生成 shim 的脚本，但由于 Python 的 `http.server` 在通过 Playwright `webServer` 生命周期管理时，难以稳定支撑 Playwright 的高并发连接（导致随机的超时与挂起），导致脚本运行不稳定。
- **退回方案 A**：根据指令，若脚本无法做到稳定，则退回方案 A，不引入任何仓库内的脚本。
- **关于临时 shim**：
  - 为什么需要 shim：本地安装的 bundled playwright 采用了 pnpm-style 的依赖布局，导致直接执行 node.exe 时 Node 无法解析 `@playwright/test` 的包路径，必须依赖一个本地文件夹作为 shim 来进行 `module.exports = require('playwright/test')` 重定向。
  - shim 在仓库外：该 shim 完全存放于系统 `%TEMP%` 目录，并未进入代码仓库。
  - 如何避免提交 shim：由于 shim 位于仓库目录之外，Git 并不会跟踪，自然规避了提交风险。
  - **推荐修复方式**：完全修复此问题的正式途径，是修复宿主机/Agent 运行环境中的 `PATH` 环境变量，让 `npm` 可见；或者重新通过规范的 `npm install` 安装完整依赖树。
- **当前 Fallback 命令完整步骤**：
  1. 打开独立终端执行：`py -m http.server 8000` （让单线程静态服务器提前、独立启动）
  2. 创建临时目录 `$env:TEMP\d3-playwright-shim\@playwright\test` 并在其中写入映射 `index.js` 和 `package.json`。
  3. 执行：
     ```powershell
     $env:NODE_PATH="C:\Users\zhaob\AppData\Local\Temp\d3-playwright-shim;C:\Users\zhaob\.cache\codex-runtimes\codex-primary-runtime\dependencies\node\node_modules\.pnpm\node_modules;C:\Users\zhaob\.cache\codex-runtimes\codex-primary-runtime\dependencies\node\node_modules"
     C:\Users\zhaob\.cache\codex-runtimes\codex-primary-runtime\dependencies\node\bin\node.exe C:\Users\zhaob\.cache\codex-runtimes\codex-primary-runtime\dependencies\node\node_modules\playwright\cli.js test --project=chromium
     ```
