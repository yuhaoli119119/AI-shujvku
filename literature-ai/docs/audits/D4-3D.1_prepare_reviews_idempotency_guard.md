## 3. Endpoint Idempotency Finding

Before this gate:

- Service lookup already used `(paper_id, target_type, target_id, field_name)`, so exact duplicate rows were unlikely.
- The API returned only a list of reviews, so callers could not distinguish created vs existing/skipped rows.
- Re-running prepare on an existing non-verified row could refresh mutable review fields and force status back to pending.

After this gate:

- First prepare creates missing review rows as `reviewer_status=pending`.
- Second prepare returns `created_count=0` and reports existing rows.
- Existing verified rows are skipped and not downgraded.
- Existing non-verified rows are not duplicated and are not upgraded to verified.
- Existing non-verified rows only receive missing prepare metadata/evidence fields when blank.
- The endpoint response includes:
  - `created_count`
  - `existing_count`
  - `skipped_count`
  - `review_ids`
  - `verified_count`
  - `safe_verified_count`
  - `items`

The endpoint does not accept or trust `reviewer_status=verified` or `verified=true` in request payloads.

## 9. Recommendation For Antigravity D4-3D.2

Add a frontend prepare entry only behind an explicit user action. The UI should show the prepare summary (`created_count`, `existing_count`, `skipped_count`, `verified_count=0`, `safe_verified_count=0`) and must prove that merely opening Human Workbench never calls prepare.
