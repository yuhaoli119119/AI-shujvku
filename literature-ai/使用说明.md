# Literature AI 使用说明

本文面向日常使用者，描述当前可用入口、启动方式和常见问题。当前定位是 Codex 使用的本地文献工具台：软件负责入库、解析成可读中间产物、检索证据和保存候选；最终阅读、核对、总结、写作和数据整理由 Codex 或人工完成。

## 1. 启动服务

在项目目录执行：

```powershell
cd literature-ai
Copy-Item .env.example .env
# 先把 .env 中的 PostgreSQL、MinIO、Owner、MCP 占位密钥替换为随机值
docker compose up --build
```

常见容器：

- `backend`
- `worker`
- `redis`
- `minio`
- `grobid`
- `postgres`（主数据库，必须启动）

## 2. 健康检查

启动后访问：

- <http://localhost:8000/api/health>

如果返回健康状态，说明后端已经可用。

## 3. 常用入口

- 文献库工作台：<http://localhost:8000/pages/literature_library/index.html>
- 入库中心：<http://localhost:8000/pages/ingestion/index.html>
- DFT 数据库：<http://localhost:8000/pages/dft_database/index.html>
- 写作辅助：<http://localhost:8000/pages/writing_assistant/index.html>
- 论文详情页：`http://localhost:8000/pages/paper_detail/index.html?id=<paper_id>`

历史入口如外部 AI 工作台、AI Writer、机理知识聚合仍可直接访问，但不再作为主流程入口。

## 4. 文献库怎么用

在文献库工作台中，你可以：

- 上传本地 PDF
- 按 DOI 下载并入库
- 做在线文献搜索或辅助批量检索
- 查看论文详情、Sections、Tables、Figures
- 查看 DFT、机理、电化学、Writing Cards 等候选抽取结果
- 导入网页 AI、外部模型或人工整理的解析候选，供 Codex 后续整理

当前活跃文献库的数据会写入 PostgreSQL 数据库中。

如果你使用项目自带的 Docker Compose，宿主机上的默认位置是：

- `literature-ai/data/library_registry.json`
- `literature-ai/data/storage/` (PDF 和图片等文件存储)

## 5. 论文详情怎么看

论文详情页或主工作台详情面板通常可查看：

- 论文基础元数据
- 原文分段与结构
- 抽取出的结构化字段
- 部分证据定位与 PDF 预览能力

如果 PDF 缺失，系统会提示无法预览，而不是自动补全不存在的文件。

## 6. Codex / MCP 怎么用

入口：

- 文献库中的“解析导入与核对”页签
- MCP 地址：`http://localhost:8000/mcp`
- 连接指南：<http://localhost:8000/api/system/agent-guide>

典型流程：

1. 在文献库中选择论文，查看摘要、章节、图表、DFT 候选和中文译文。
2. Codex 优先通过 MCP 读取 `query_papers`、`get_paper`、`retrieve_evidence` 等结果。HTTP MCP 始终需要 `LITAI_MCP_API_KEYS` 中配置的 Bearer Key，不因本机或私网 IP 自动授权。如果当前 IDE 会话没有暴露 MCP 工具，也可以直接走仓库内 `literature-ai/backend` 的 `app.mcp.context.mcp_auth_context` + `app.mcp.server` 后备路径。
3. 优先调用 `get_codex_context` 或 `GET /api/papers/{paper_id}/codex-context` 获取单篇文献的压缩上下文包。
4. 如果 PDF 太长或 token 成本高，可以先用网页 AI / 外部模型生成解析文本或 JSON。
5. 将外部解析结果导入为候选。
6. Codex 根据原文、证据、候选和你的要求整理归纳。
7. 需要入库修正时，先提出 correction / note，再人工确认。

注意：网页 AI、自动解析、外部模型生成内容都只是候选，不等于 verified。

## 7. Docker 相关常见问题

如果页面打不开，先检查容器是否启动：

```powershell
docker compose ps
```

如果后端未正常启动，查看日志：

```powershell
docker compose logs backend --tail=200
```

## 8. 常见问题

### 如何分享给外部用户查看？

现在系统支持只读分享链接。具有独立 `create_share_links` capability 的 MCP Key 可以调用 `create_share_token` 生成链接。Docker Compose 默认把本机 Owner 入口绑定到 `127.0.0.1:8000`，把 LAN 只读分享入口绑定到 `0.0.0.0:8080`；分享链接形如 `http://<局域网IP>:8080/pages/share/index.html?token=xxx`。分享入口只代理分享页面、`/api/share/*` 和健康检查，不能访问管理 UI、普通 API 或 MCP。

### 为什么导出按钮返回 403？

`LITAI_EXPORTS_ENABLED` 默认是 `false`，会统一关闭 CSV、DFT dataset、writing export、Word 文件生成/下载和 MCP `export_ml_dataset`。需要恢复时必须由 Owner 明确改为 `true`，并为 MCP Key 单独授予 `export_data` capability。

### 本地路径入库允许哪些目录？

`LITAI_LOCAL_INGEST_ROOTS` 是逗号分隔的允许根目录。同步接口、后台 job 和 MCP 批量入库共用同一校验：文件必须位于允许根下、是普通 `.pdf` 文件，并具有 `%PDF-` 文件头。

### 能不能手动清理 artifacts？

默认不要。除非任务明确要求，并且先确认影响范围。
